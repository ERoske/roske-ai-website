Architecture

One engine, layered for clarity.

AgentCube sits between your AI agent and your Oracle systems. Everything above the engine is pluggable (which AI, which transport); everything below is configurable (which Oracle systems, which auth).

Transports

Three ways to serve. Pick based on where your agent runs.

AgentCube supports the three transports defined by the Model Context Protocol. All three ship in the box; switching is a single CLI argument.

--transport stdio

STDIO

Process-pipe transport. The agent launches AgentCube as a subprocess and communicates via stdin/stdout.

Use when: running locally alongside a desktop AI client (Claude Desktop, local Copilot, Cursor). Zero network surface.

--transport sse

SSE

Server-Sent Events transport. AgentCube runs as a long-running HTTP service; agents connect and subscribe to a streaming endpoint.

Use when: single-tenant production deployments with a small number of concurrent agent sessions.

--transport streamable-http

Streamable HTTP

Modern MCP transport. Bi-directional streaming over HTTP/2. Scales horizontally, plays nicely with standard load balancers and ingress.

Use when: multi-tenant production, enterprise deployments with load balancers, or behind standard API gateways.

Implementation

A complete MCP server in 15 lines.

A minimal AgentCube-backed MCP server that exposes Essbase application listing to any compatible AI agent. This is the real thing, not an excerpt.

# Complete MCP server for Essbase using AgentCube
from fastmcp import FastMCP
from agentcube import AgentCube
from agentcube.providers.essbase import EssbaseProvider
from agentcube.config.essbase import EssbaseConfig

mcp = FastMCP("Finance Ops Agent")
engine = AgentCube()
engine.register(EssbaseProvider(EssbaseConfig.from_env()))

@mcp.tool()
def list_applications() -> dict:
    """List all Essbase applications available to the user."""
    return {"apps": engine.list_applications("essbase")}

if __name__ == "__main__":
    mcp.run(transport="stdio")  # or "sse", "streamable-http"

Swap in PlanningProvider or FCCProvider (or all three) to serve any combination. Mix providers to build domain-specific agents.

Specifics

The things you'd only ask if you were actually going to deploy this.

01 · Runtime

Python 3.11+, installed as a package.

AgentCube is distributed as a standard Python package. Install with pip install agentcube or via a private artifact in air-gapped environments. Dependencies are minimal: FastMCP, httpx, pydantic. No framework lock-in. No heavyweight runtime.

02 · Configuration

Environment variables, config files, or programmatic.

Each provider takes a configuration object that can be built from environment variables (for simple deployments), a YAML or JSON config file (for declarative setups), or programmatically (for integration into your existing service scaffolding).

  • ESSBASE_SERVER_URL, ESSBASE_USERNAME, ESSBASE_PASSWORD, ESSBASE_AUTH_TYPE
  • PLANNING_SERVER_URL, PLANNING_USERNAME, PLANNING_IDENTITY_DOMAIN
  • FCC_SERVER_URL, FCC_USERNAME, FCC_IDENTITY_DOMAIN
03 · Auth Patterns

Delegated auth, never stored in the engine.

AgentCube does not store credentials. It accepts them per-session from the caller (via config or OAuth token) and uses them to call Oracle. Session tokens are held in memory only. Add a secrets manager (AWS Secrets Manager, Azure Key Vault, HashiCorp Vault) for persistence; AgentCube has hooks for all three.

  • Basic: HTTPS-only, best for air-gapped or development
  • OAuth 2.0: Client credentials or authorization code flow against Oracle IDCS
  • OIDC / SSO: Federated identity through your enterprise IDP
04 · Tool Schemas

Every tool call is typed, validated, and auditable.

AgentCube tools are defined with pydantic models. Every argument is typed; every response schema is known. When an AI agent tries to call a tool with bad arguments, the schema catches it before the call reaches Oracle. When an agent gets a response, it gets structured data, not freeform text it has to parse.

This matters for audit: you can reconstruct every interaction from logs because you know exactly what the schema required and what was passed.

05 · Custom Providers

Extend to your stack in under a day.

A provider is a Python class that implements three things: a config dataclass, an authenticated HTTP client, and a set of tool methods. Add a new provider for your custom EPM extension, your data warehouse, or your home-grown reporting system, and it composes with the Oracle providers automatically.

Certified Caprus partners build custom providers for client-specific extensions regularly. Partner SDK and example providers available under NDA.

06 · Observability

JSON logs, OpenTelemetry traces, Prometheus metrics.

AgentCube emits structured logs to stdout by default. Every session, tool call, and response is logged with identity, timestamps, and metadata (no financial values). Turn on OpenTelemetry tracing to see end-to-end spans from agent prompt through Oracle response. Turn on Prometheus metrics to monitor latency, throughput, and error rates.

  • agentcube.tool.duration_ms: histogram of call latency by provider and function
  • agentcube.tool.errors_total: counter of errored calls by error class
  • agentcube.session.active: gauge of currently active sessions
07 · Performance

Bound by Oracle, not by AgentCube.

Tool call latency is dominated by the underlying Oracle API. AgentCube itself adds a few milliseconds of dispatch, validation, and logging overhead per call. A typical Planning dimension query returns in 200-800ms; Essbase MDX queries take 500ms to several seconds depending on cube complexity; FCC consolidation status queries return in under a second.

For bulk scenarios, use batching or parallel tool calls from the agent. The engine handles concurrent sessions cleanly; scale horizontally for more throughput.

Before You Deploy

The pre-deployment checklist your EPM admin will ask about.

If you can check these boxes, you can deploy AgentCube in an afternoon. If you can't, you know exactly what to line up first.

You need:

  • An Oracle EPM system we support (Essbase, Planning, or FCC). Modern Oracle Cloud or on-premises 11.2+.
  • Service account or OAuth application configured in your Oracle identity system with API access.
  • Network path from wherever AgentCube runs to your Oracle APIs (internal network, VPN, or VPC peering).
  • A host with Python 3.11+, 2 vCPU, 2 GB RAM minimum. Container-friendly; runs fine in a small pod.
  • Your AI agent configured to speak MCP. Claude Desktop, Cursor, and Copilot do this natively; others need a thin client.
  • A standard logging destination (stdout capture, journald, a SIEM, or a log aggregator).
