The Specifics

How every security control actually works.

Every area your CISO will ask about, answered before the security review meeting.

01 · Identity

Authentication to Oracle: your existing mechanism, no exceptions.

AgentCube does not issue credentials to Oracle systems. It delegates authentication to whatever your enterprise already uses. Every provider accepts:

  • Basic for local development or air-gapped deployments with service accounts
  • OAuth 2.0 for modern token-based authentication against Oracle Identity Cloud Service
  • OIDC / SSO for federated identity through Okta, Azure AD, Ping, or your custom IDP

The agent acts on behalf of the signed-in user. Tokens are per-session, scoped to the minimum set of Oracle permissions required, and never persisted beyond the session unless you explicitly configure a secure vault.

02 · Authorization

Oracle's existing access model is the boundary. We don't widen it.

Your Planning, Essbase, and Financial Close systems already enforce row-level security, entity-level access, and cube-level permissions. AgentCube queries through those systems using the authenticated user's identity. If the user can't see Germany's Q3 numbers in Smart View today, they can't see them through an AgentCube-powered agent tomorrow.

This has a useful side effect: your existing audit and SOX controls keep working. You are not introducing a new entitlement surface to govern, just a new query interface over the surface you already govern.

03 · Isolation

Multi-tenant by design, single-tenant by default deployment.

Sessions are isolated at the process or connection level, depending on transport. Configuration, tool schemas, credentials, and pending tool-call state never cross between sessions. When you deploy AgentCube for your organization, you can run it single-tenant (one instance per business unit) or multi-tenant (one instance serving many customers with enforced boundaries). The isolation guarantees hold either way.

No shared caches hold financial values between sessions. No background workers fan out across tenants. No cross-tenant telemetry.

04 · Observability

Every tool call is a structured audit event.

AgentCube emits a structured log entry for every:

  • Authenticated session (who, when, from where)
  • Tool call (which provider, which function, which arguments)
  • Response summary (status code, byte counts, error classes)
  • Configuration change (who changed what, when)

Logs are JSON, stream to stdout by default, and can be routed to Splunk, Datadog, Elastic, or any SIEM that ingests JSON lines. Business values (actual financial figures) are not included in logs unless you explicitly opt in for debugging, which we advise against in production.

05 · Data Handling

AgentCube is a router. It does not store your financial data.

Financial data is pulled from Oracle on demand, passed to the AI agent, and released. There is no AgentCube data warehouse, no training corpus, no retention.

Whether the data reaches an LLM hosted by OpenAI, Anthropic, Google, Microsoft, or your own private model is entirely governed by the agent you connect. AgentCube is the integration layer. If your policy is "no data leaves our tenant," point AgentCube at a model running in your tenant and the policy holds.

06 · Deployment

Run it where your data already lives.

AgentCube is a Python package plus a set of thin server wrappers. Deploy patterns:

  • On-premises alongside your Oracle EPM infrastructure, no outbound internet required
  • Private cloud in your AWS, Azure, or OCI tenant
  • Hybrid with the engine in your cloud and agents in a customer tenant
  • Managed by Caprus as a hosted service (select customers only)

Transports include STDIO (process-pipe for local development), SSE (server-sent events), and Streamable HTTP (for production). Switch via a single CLI flag.

07 · Compliance

Designed to fit into your existing program.

Because AgentCube runs in your environment and defers authentication and authorization to your Oracle systems, it inherits most of your existing compliance posture. The AgentCube layer itself is built for:

  • SOC 2 Type II alignment for the managed service offering
  • GDPR posture through no-retention-by-default and identity-scoped access
  • SOX audit support via immutable structured logs tied to user identity
  • HIPAA readiness for finance teams in regulated healthcare settings (on-prem deployment)

We provide a security questionnaire (CAIQ and SIG Lite available) and a dedicated security review call for enterprise evaluations.

Security Questions

The questions every CISO asks first.

Does our financial data ever leave our tenant?
Only if you explicitly configure it to. AgentCube is the connector between an AI model and your Oracle systems. If you point it at a model running in your private cloud or on-premises, data stays in your tenant end to end. If you point it at a third-party model API, the data traverses whatever boundary that model requires. Your data egress policy governs, not ours.
How do you handle authentication with Oracle?
We don't create new Oracle credentials. AgentCube accepts whatever your Oracle systems accept (Basic, OAuth 2.0, OIDC/SSO) and delegates auth to your identity provider. The agent calls Oracle under the authenticated user's identity, scoped to the permissions that user already has.
What's logged, and can we stream it to our SIEM?
Structured JSON logs are emitted for every authenticated session, every tool call, every response status, and every configuration change. Logs include user identity, timestamps, provider, function, and metadata. Business values (actual financial figures) are excluded by default. Logs stream to stdout and can be routed to Splunk, Datadog, Elastic, or any JSON-ingesting SIEM.
Is there a data retention policy we need to worry about?
AgentCube does not retain business data. Financial figures flow through the engine on demand and are returned to the caller. No caching of values between sessions. No training corpus. No background processing. The only retention is the metadata in audit logs, and you control how long those are kept.
How do you prevent prompt injection from exfiltrating our data?
Prompt injection is a model-layer concern, not a connector-layer concern, but AgentCube includes several mitigations: tool calls are scoped by authenticated user (injection can only access what that user can access); every call is logged (exfiltration attempts are visible in audit); tool argument schemas are enforced (agents can't pass arbitrary SQL or payloads). Your model layer should add input sanitization and output moderation.
Do you have SOC 2, GDPR, or SOX documentation?
Our managed service offering is SOC 2 Type II aligned; documentation is available under NDA during enterprise evaluations. GDPR posture is supported through identity-scoped access and no-retention defaults. SOX audit support is built in through immutable structured logs. We maintain CAIQ and SIG Lite questionnaires and make our security team available for review calls.
Can we run AgentCube air-gapped?
Yes. Air-gapped deployment is fully supported. AgentCube is a Python package; installation and updates can be delivered via offline artifacts. Pair it with an on-premises model (Llama, Mistral, or your own fine-tuned model) and you have a fully self-contained AI-over-EPM environment with zero outbound connectivity.
What happens if AgentCube hallucinates or fabricates a number?
AgentCube does not generate numbers. It retrieves them from Oracle via official APIs and returns the structured response. The AI model that uses AgentCube can summarize or explain, but the figures themselves come from Oracle. If a finance team asks "what was Germany Q3 revenue," the answer is the value Oracle returned, not a model's best guess. Every call is logged and reproducible.
